第一章 绪论与系统综述1.1 研究背景与目标概述随着网络音乐互动平台的兴起，以 VirtualPiano.net、AutoPiano.cn 以及 Roblox 游戏中的钢琴模块为代表的“虚拟钢琴”生态系统，已经发展成为一种独特的数字音乐亚文化。这类平台的核心交互机制在于将计算机的标准 QWERTY 键盘映射为钢琴的琴键，使用户能够通过敲击键盘上的字母和数字来模拟乐器演奏。为了降低演奏门槛，通过算法将现有的音频文件（MP3, WAV）自动转换为该平台专用的“文字乐谱”（Text Sheet Music），成为了技术社区与音乐爱好者共同关注的焦点。本报告旨在响应制作一款“音频转文字乐谱”程序的开发需求。该程序的核心任务是建立一条高精度的转换管道，能够接收用户上传的复杂音频信号，经由信号处理与深度学习模型解析，最终输出符合 AutoPiano.cn 或 VirtualPiano 格式规范的字符序列。用户依据此序列，即可利用键盘脚本（如 AutoHotkey）或手动输入，实现对原始音乐的高度还原演奏。1.2 核心技术挑战实现这一目标并非单纯的格式转换，而是跨越了音频信号处理（Audio Signal Processing）、音乐信息检索（Music Information Retrieval, MIR）与自然语言处理（NLP）边界的综合性工程。其核心难点主要集中在以下三个维度：声源分离的纯净度：用户上传的音频往往是混音后的成品（Full Mix），包含人声、鼓点、贝斯等多种乐器。传统的频率分析方法（如傅里叶变换）无法在复杂的频谱中精准剥离出钢琴信号，导致非钢琴声音被误识别为音符，严重影响转换精确度。复调音乐转录的准确性：钢琴作为多声部（Polyphonic）乐器，往往同时存在多个音符（和弦）。不同音符的泛音列（Harmonics）会在频域上发生重叠，使得基于规则的基频检测算法失效。此外，钢琴特有的延音踏板（Sustain Pedal）会模糊音符的起止时间，增加了节奏量化的难度。QWERTY 映射的物理约束：MIDI 协议支持 128 个音符与任意的复调，但计算机键盘存在“键位冲突”（Ghosting）与布局限制。例如，同一个键位在按下 Shift 键时代表不同的音高，如何在文本脚本中处理“同时也按住 Shift 和不按 Shift 的键”是逻辑转换层必须解决的难题。1.3 报告结构本报告将分章节详细阐述实现该系统的完整技术路径。第二章将深入解析虚拟钢琴的符号系统与脚本规范；第三章与第四章将分别探讨利用深度学习进行声源分离（Source Separation）与自动音乐转录（AMT）的最前沿方案，重点对比 ByteDance、Google Magenta 与 Spotify 的模型架构；第五章将详细论述从 MIDI 数据到文字乐谱的算法映射逻辑；最后，第六章将针对“提高精确度”这一核心诉求，提出包括数据增强、频谱通量校正及交互式编辑在内的高级优化策略。第二章 虚拟钢琴符号系统与规范解析在构建算法之前，必须对目标输出格式——即虚拟钢琴的“文字乐谱”——进行严谨的定义与解析。不同于五线谱的视觉化语言或 MIDI 的二进制指令，虚拟钢琴乐谱是一种基于 ASCII 字符的直观记谱法。2.1 键位映射协议（Key Mapping Protocol）虚拟钢琴的核心机制是将钢琴的 88 个琴键（或 61 键）映射到计算机键盘的四排按键上。这种映射通常遵循“从左至右、从下至上”的半音阶排列逻辑。2.1.1 标准化映射逻辑目前主流平台（包括 AutoPiano.cn）采用的映射标准涵盖了从低音到高音的连续区间。以标准的 61 键布局为例，映射通常始于 C2 或 C3 音域。白键映射：通常对应键盘上的数字行与字母行的小写状态。例如，1, 2, 3 对应 C, D, E 等低音区白键；q, w, e 对应中音区白键。黑键映射（升降号）：这是转换程序的逻辑难点。黑键通常映射为对应白键的“Shift”状态（即大写字母或符号）。例如，若 s 键映射为 D（Re），则 S（即 Shift + s）通常映射为 D#（升 Re）。数字键的黑键则对应其特殊符号，如 1 为 C，!（Shift + 1）则为 C#。表 2.1：典型 MIDI 音高与虚拟钢琴字符映射对照表（部分）MIDI 音高编号科学音高记号虚拟钢琴字符 (不按Shift)虚拟钢琴字符 (按住Shift/黑键)备注48C31!低音区起始49C#3-!需组合键50D32@-52E33-无黑键53F34$-60C4 (中央C)8*中音区61C#4-*-72C5tT高音区注：不同平台（如 字符本身不携带时间信息，虚拟钢琴社区发展出了一套基于“空格”与“分隔符”的伪节奏系统。2.2.1 相对时间间隔程序必须解析 MIDI 时间戳（Delta Time），并将其量化为以下符号：无空格连写（如 asdf）：代表极短的时间间隔，通常用于表示 16 分音符、32 分音符的快速跑动，或是装饰音、滑音（Glissando）。在自动脚本中，这通常意味着每个按键之间的延迟（Delay）设定为极小值（如 < 50ms）。单个空格（如 a s d f）：代表标准的旋律进行速度。空格本身充当了一个“单位延时符”。在 4/4 拍的乐曲中，这可能对应一个 8 分音符的长度。和弦括号（如 [asdf]）：这是复调音乐的核心。被方括号 `` 包裹的字符必须同时被按下。算法在处理时，必须检测同一时间戳（或极短时间窗内，如 30ms）的所有 Note On 事件，并将它们聚合为一个字符组。2.2.2 停顿与小节线竖线 |：通常用于表示较明显的停顿，或者作为小节线（Bar Line）的视觉参考。在自动播放脚本中，| 往往被硬编码为一个较长的延时（例如 500ms 或一个四分音符的时值）。复合停顿：如 | | 或 |，表示更长的休止符。程序在转换时，需要根据检测到的 BPM（Beats Per Minute）计算休止符的时长，并插入相应数量的 | 符号。2.3 脚本格式的多样性用户提到的 autopiano.cn 既支持直接的文字输入，也常与自动化脚本工具（如 AutoHotkey - AHK）配合使用。生成的程序应具备两种输出模式：纯文本模式（Raw Text）：仅输出 [uo] a [sp]...，供用户复制粘贴到网页输入框。脚本代码模式（Script Mode）：直接生成 .ahk 或 Python 脚本，调用底层键盘驱动（如 pynput 或 SendInput API）模拟按键。这种模式能更精确地控制按键时长（Duration），从而模拟钢琴的延音效果，而纯文本模式通常只能触发“按下”动作，无法控制“松开”的时机。第三章 音频信号处理与声源分离技术为了实现“提高转换精确度”的目标，直接对原始音频进行转录是不可取的。现代音乐录音通常是混合了多种乐器的复杂信号，如果直接将一首包含人声和鼓点的流行歌曲输入到钢琴转录模型中，模型会将人声的基频误判为高音区琴键，将鼓点误判为低音区重击。因此，声源分离（Source Separation） 是构建高精度系统的第一道必经工序。3.1 频谱混叠与传统方法的局限在深入深度学习方案前，有必要理解为何传统的信号处理方法（如 FFT）无法胜任此任务。傅里叶变换（FFT）能将时域信号转换为频域信号，显示特定时刻的频率成分。然而，钢琴的一个音符并非单一的正弦波，而是由基频（Fundamental Frequency）和一系列整数倍的泛音（Overtones）组成的复合音。泛音重叠：当演奏 C4（261.6Hz）和 C5（523.2Hz）八度双音时，C4 的第二泛音与 C5 的基频完全重合。FFT 无法区分这股能量是来自 C5 还是 C4 的泛音。瞬态模糊：钢琴的音头（Attack）包含宽频带的敲击声，FFT 在保证频率分辨率的同时会损失时间分辨率（海森堡测不准原理），导致无法精确捕捉音符开始的毫秒级瞬间。3.2 深度学习声源分离架构：Demucs目前（2024-2025年），在声源分离领域占据统治地位的是由 Meta Research (Facebook) 开发的 Demucs 模型，特别是其 Hybrid Transformer Demucs (htdemucs) 变体。3.2.1 架构优势Demucs 采用了一种混合架构，结合了时域（Time Domain）上的卷积网络与频域（Frequency Domain）上的频谱分析，并引入了 Transformer 注意力机制。波形域操作：不同于早期的 Spleeter（基于频谱掩码），Demucs 直接在波形上进行端到端的预测。这使得它能保留高频细节（最高可达 44.1kHz 的奈奎斯特频率），而 Spleeter 往往在 11kHz 或 16kHz 处有一刀切的截止，导致钢琴的高音部分听起来沉闷甚至丢失。相位重建：在钢琴转录中，相位信息包含了音色的微细变化。Demucs 能够更好地重建相位，使得分离出的钢琴音轨（Stem）更加清晰，减少了“起伏噪声”（Musical Artifacts）。3.2.2 与 Spleeter 的对比评估在构建本程序时，强烈建议集成 Demucs 而非 Spleeter。表 3.1：声源分离模型对比分析特性维度Spleeter (Deezer)Hybrid Demucs (Meta)对钢琴转录的影响工作域仅频域（频谱掩码）时域 + 实施策略在程序设计中，应包含一个预处理模块：接收用户上传的 input.mp3。调用 demucs 命令行工具或 Python API：demucs -n htdemucs_ft "input.mp3"。系统自动提取分离出的 other.wav（通常包含钢琴）或在四轨分离模式下的 piano.wav（如果模型支持特定乐器分离）。将该纯净的 WAV 文件作为下一阶段转录引擎的输入。第四章 核心转录引擎：自动音乐转录 (AMT)在获得纯净的钢琴音频后，下一步是将波形转换为符号化的 MIDI 事件（Note On, Note Off, Velocity）。这是系统的“大脑”。为了达到用户要求的“高精确度”，必须选用当前学术界最先进（State-of-the-Art, SOTA）的模型。4.1 字节跳动（ByteDance）高分辨率钢琴转录模型根据最新的文献综述，由字节跳动 AI Lab 发布的 High-Resolution Piano Transcription with Pedals 是目前钢琴转录领域的标杆。4.1.1 技术原理：回归优于分类传统的 AMT 模型（如 Google 的 Onsets and Frames）通常将转录视为一个分类问题：将频谱图划分为一个个 32ms 的帧，判断每一帧里哪个音符是“开启”的。这种方法的局限在于时间精度受限于帧的大小，导致音符时值不够精确。字节跳动的模型采用了一种**回归（Regression）**策略：连续时间预测：它不仅仅预测哪一帧有音符，还预测音符在帧内的精确偏移量（Offset）。这使得它能以极高的时间分辨率（High-Resolution）定位音符的起始和结束。速度检测（Velocity Regression）：该模型能精准预测击键力度（0-127）。虽然虚拟钢琴脚本通常不支持力度回放，但这一信息对于过滤“幽灵音符”（Ghost Notes，即误识别的极弱噪音）至关重要。4.1.2 踏板处理（Pedal Transcription）这是该模型最大的杀手锏。钢琴演奏中大量使用延音踏板，这会使得声音持续时间远长于手指按键时间。问题：普通模型会把延音听成是手指一直按着键，导致生成的乐谱充满了长音，在虚拟钢琴上回放时会占用按键通道，导致后续音符无法弹奏。解决方案：ByteDance 模型有专门的分支网络检测踏板的 On/Off 事件。这使得程序可以区分“手指松开但踏板踩下”的状态，从而在生成文字乐谱时，正确地断开字符，避免不必要的长按指令。4.2 替代方案：Spotify Basic Pitch如果用户的硬件配置较低（无独立显卡），或者需要转录非钢琴乐器（如吉他独奏）作为钢琴谱，Spotify Basic Pitch 是一个优秀的轻量级备选。架构：基于轻量级卷积神经网络（CNN），针对多种乐器训练。特点：支持弯音（Pitch Bend）检测。虽然钢琴没有弯音，但如果用户上传一段小提琴音频想转成钢琴谱，Basic Pitch 能捕捉到频率的滑动，尽管转换为固定音高的钢琴键时需要进行量化处理。4.3 Google Magenta (Onsets and Frames)这是上一代的 SOTA 模型，基于 Bi-LSTM（双向长短期记忆网络）。现状：虽然依然强大，但在处理短促音符的断开和踏板共鸣方面，已被 ByteDance 的架构超越。除非需要完全的 TensorFlow 生态支持，否则在 2025 年的节点上，建议优先选择 ByteDance 的 PyTorch 实现。表 4.1：转录模型性能指标对比 (基于 MAESTRO 数据集)模型Note On F1 Score (准确率)Note 文件后，真正的挑战才刚刚开始。MIDI 是机器的语言，而 autopiano.cn 需要的是特定的字符序列。这一章节涵盖了程序开发中逻辑最复杂的部分：量化（Quantization）与 映射（Mapping）。5.1 时间轴的网格化（Quantization）AI 生成的 MIDI 音符时间是浮点数（例如：Start: 1.2345s, End: 1.5678s）。直接转换会导致极其混乱的节奏。5.1.1 动态 BPM 检测与网格对齐程序需要首先利用 librosa.beat.beat_track 分析音频的 BPM。假设检测到 BPM 为 120。计算 Tick 时长：在 16 分音符的精度下，一个 Tick = $60 / 120 / 4 = 0.125$ 秒。吸附（Snapping）：将所有音符的 Start Time 四舍五入到最近的 Tick 整数倍。这能将微小的演奏误差（Timing Jitter）修正为标准的节奏。5.1.2 和弦聚合窗口在虚拟钢琴中，[abc] 表示同时按下。但在 MIDI 中，即便是和弦，不同音符的 Note On 时间也可能相差几毫秒。算法逻辑：设定一个聚合窗口（例如 30ms - 50ms）。遍历时间轴，如果 Note B 的开始时间在 Note A 开始后的 30ms 内，则判定它们属于同一个“和弦事件”，在输出文本时将其包裹在同一个 `` 中。5.2 键位映射引擎（The Mapper Engine）这是连接 MIDI 音高与 QWERTY 字符的桥梁。5.2.1 字典构建程序需内置一个完整的映射字典（Dictionary）。根据 autopiano.cn 的常用脚本规范，键位表如下：Python# 示例映射表（部分），实际需覆盖 MIDI 36 (C2) 到 5.2.2 冲突消解：Shift 键的物理悖论这是提高精确度的关键细节。场景：乐曲中出现了一个和弦，包含 MIDI 72 (C5 -> 't') 和 MIDI 73 (C#5 -> 'T')。冲突：在物理键盘上，用户无法同时输入 't'（小写）和 'T'（大写），因为 Shift 键是全局生效的。如果脚本发送 Shift + t，结果是 T；如果不按，结果是 t。算法策略：检测冲突：检查同一个和弦括号 `` 内是否同时存在即需 Shift 又不需 Shift 的字符。琶音化处理（Arpeggiation）：将冲突的和弦拆分为两个极短间隔的序列。例如，将 `` 转换为 t (短延迟) T。虽然听起来略有不同，但保证了物理可执行性。异名同音转换：在某些复杂的脚本映射中，可能存在替代键位。程序应搜索是否存在不冲突的替代键。5.3 复调限制与可玩性优化（Playability Optimization）为了确保输出的乐谱是“可弹奏”的（无论是人手还是脚本），必须进行复调削减。浏览器限制：网页版钢琴通常对同时按下的键数有限制（N-key Rollover 问题）。算法：对于任何一个时间点，如果音符数量超过 N（例如 5 个），程序应依据**速度（Velocity）**保留最大的 N 个音符，或者优先保留根音（最低音）和最高音（旋律线），剔除中间的填充音，以保证主旋律的清晰度。第六章 提高转换精确度的进阶策略响应用户“如何提高转换精确度”的提问，本章提供超越基础实现的优化方案。6.1 基于频谱通量的音头校正（Onset Correction）AI 模型偶尔会发生时间上的“漂移”。方法：结合传统的 DSP 方法。使用 librosa.onset.onset_detect 计算频谱通量（Spectral Flux）的峰值。融合：如果 AI 预测音符在 1.05秒开始，但 DSP 检测到 1.02秒有一个强烈的能量突发，强制将音符对齐到 1.02秒。这种“AI + DSP”的混合方法能显著提升节奏的紧凑感。6.2 动态阈值过滤（Velocity Thresholding）AI 模型往往极其敏感，会捕捉到录音中的背景底噪或钢琴内部的机械共振，生成力度极小（Velocity < 20）的杂音。策略：设置一个动态阈值（例如：Threshold = Average_Velocity * 0.2）。低于该阈值的音符直接丢弃。这能使生成的乐谱瞬间变得干净清爽，去除那些人类听不到的“幽灵音符”。6.3 交互式“钢琴卷帘”编辑器（Human-in-the-Loop）无论算法多完美，总有误判。提高最终精确度的终极手段是引入人工修正。功能设计：在程序中集成一个可视化的钢琴卷帘（Piano Roll）界面。用户可以在导出文字乐谱前，直观地看到 MIDI 音符，并手动删除错误的音符、调整时值或修正错位的节奏。这是所有专业级转录软件（如 Melodyne）的标准功能。第七章 系统实现架构与技术栈推荐为了将上述研究转化为实际程序，以下是推荐的技术栈与开发路线图。7.1 技术栈选择编程语言：Python 3.9+ （拥有最丰富的音频处理生态）。音频加载与分析：librosa, soundfile。AI 框架：PyTorch (用于运行 Demucs 和 ByteDance 模型)。MIDI 处理：pretty_midi 或 mido。用户界面 (GUI)：PyQt6 或 Electron (如果做成 Web App，后端需使用 Flask/FastAPI 封装 Python 脚本)。7.2 伪代码逻辑流Pythondef main_pipeline(audio_file_path):

def convert_midi_to_vp_text(midi, mapping):
    timeline = group_notes_by_time(midi)
    output_str = ""
    last_time = 0
    
    for time_point in timeline:
        # 计算停顿
        delta = time_point - last_time
        output_str += generate_pause_string(delta, midi.bpm)
        
        # 生成按键字符
        notes = timeline[time_point]
        chars = [mapping[n.pitch] for n in notes]
        
        # 处理Shift冲突
        chars = resolve_shift_conflicts(chars)
        
        # 格式化
        if len(chars) > 1:
            output_str += f"[{''.join(chars)}]"
        else:
            output_str += chars
            
        last_time = time_point
        
    return output_str
